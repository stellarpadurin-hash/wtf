With Bitbucket Data Center and Tekton, finding the right cadence for running dependency updates is all about balancing **security posture** against **infrastructure cost and developer friction**.
If you trigger automated dependency upgrades on every single feature branch build, you will overwhelm your Tekton clusters and frustrate developers with a non-stop flood of automated Pull Requests.
The industry best practice for running a tool like Renovate or Snyk in this specific ecosystem is a hybrid model: run dependency updates as a **Scheduled Cron Job**, but run validation checks during the **PR workflow**.
Here is how you should structure the triggers across your pipeline.
## 1. Automated Remediation (Opening PRs): Run as a Scheduled Job
You should trigger the dependency update tool (e.g., Renovate CLI running inside a Tekton Task) via a **scheduled Tekton PipelineRun (Cron)**—for example, every Sunday night or daily at midnight.
### Why a Scheduled Job?
 * **Reduces Noise:** If a popular Node.js or Python package updates three times in one week, a daily or weekly schedule ensures your developers only see *one* consolidated PR rather than three separate notifications interrupting their active sprint work.
 * **Controlled Batching:** Running it off-hours ensures that your Tekton workers aren't bogged down running dependency resolution algorithms during peak development hours.
 * **Predictable Workflow:** Developers walk in on Monday morning, see the fresh batch of automated security PRs, and can allocate a clean block of time to review them.
## 2. Pipeline Validation: Run on Every Pull Request (Feature to Main)
While the scheduled job *creates* the remediation PRs, your standard Tekton CI pipeline must execute on **every Pull Request raised**—both human-created feature branches and bot-created remediation branches.
### What happens during the PR trigger?
When a dependency PR is opened against main, Bitbucket Data Center fires a Webhook to Tekton to trigger your standard verification pipeline:
 * It compiles the code (mvn clean compile, npm run build).
 * It runs the unit and integration test suites.
 * **Crucial Step:** It runs your **Checkmarx SAST/SCA scan** on the PR branch code. This ensures that the upgraded library version didn't introduce a new vulnerability or break compatibility.
If the Tekton pipeline passes, the safe "Patch/Minor" updates can be automatically merged (automerge), and the "Major" ones wait for a human review.
## 3. Why You Should NOT Trigger Remediation on Every Single Feature Build
Triggering a dependency search and upgrade on every individual developer commit or feature branch build is highly discouraged for three reasons:
 * **Race Conditions:** If five developers are working on five different feature branches, and the bot tries to update pom.xml or package.json on all of them simultaneously, you will end up with massive Git merge conflicts when trying to merge those branches back to main.
 * **Wasted Resources:** If a developer pushes code 10 times a day to their feature branch, running a heavy dependency evaluation task 10 times is a massive waste of Tekton compute power.
 * **Context Switching:** Developers don't want automated security commits pushed into their active feature branches while they are in the middle of writing local business logic.
## Summary of the Ideal Architecture
To make this seamless in your Bitbucket Data Center + Tekton setup, structure your workflow exactly like this:
| Activity | Trigger Type | What It Does |
|---|---|---|
| **Dependency Scanning & PR Generation** | **Scheduled (Cron)** *(e.g., Weekly or Daily at Night)* | Tekton cron triggers Renovate/Snyk CLI. It scans main, finds vulnerabilities, and opens fix PRs in Bitbucket. |
| **Build & Security Validation** | **PR Event Trigger** *(Feature \rightarrow Main)* | Bitbucket Webhook triggers a Tekton Pipeline. Runs tests, compiles the app, and runs Checkmarx to verify the upgrade is safe. |
| **Deployment Gate** | **Merge Event Trigger** *(Post-Merge to Main)* | Tekton builds the final production container artifact and triggers a **Wiz** scan on the registry image to ensure a 100% clean bill of health before deployment. |
By separating the **creation** of security fixes (Scheduled) from the **validation** of those fixes (PR-triggered), you maximize security automation without interrupting your engineering team's daily velocity.
