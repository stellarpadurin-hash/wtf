## Track 1 — Dependency vulnerabilities → Renovate Bot (fully automated)

These are findings where the vulnerable component is a third-party library version. Renovate Bot is the right tool because:

- It natively understands `pom.xml`, `build.gradle`, `requirements.txt`, `pyproject.toml`, `package.json`, and lock files across all three of your language stacks.
- It has built-in Bitbucket Data Center support — it connects directly to your repos, opens PRs, and respects branch protection rules without any custom glue code.
- You can feed it a vulnerability database (OSV, GitHub Advisory DB) so it specifically targets known-bad versions rather than just "latest."
- Your Tekton pipeline acts as the validation layer — it runs your build and tests on every Renovate PR so a bad upgrade never reaches `main`.

Wiz and Checkmarx still play a role here: they are your source of truth for which CVEs exist. You can configure Renovate to use its own advisory feeds, or you can export CVE findings from Wiz/Checkmarx and map them to package versions that Renovate should target. Many teams do both — Renovate catches known vulnerabilities proactively, and the scanner reports catch anything Renovate missed in its last cycle.

---

## Track 2 — Custom code vulnerabilities → Wiz/Checkmarx findings drive human remediation

These are SAST findings in your own code: SQL injection, insecure deserialization, hardcoded credentials, missing auth checks, XSS, path traversal, etc. There is no automated tool that can safely rewrite your business logic — attempting to do so would introduce more risk than it removes. The right workflow is:

- Wiz/Checkmarx produce a finding with file name, line number, CWE category, and severity.
- Your Tekton pipeline or a webhook automatically creates a Jira ticket (or a PR comment on the offending branch) with that detail pre-populated.
- The ticket is assigned to the team owning that service, with an SLA tied to severity (e.g. Critical = 7 days, High = 30 days).
- The fix is code-reviewed and tested normally through your existing PR → `main` flow.

---

## Where the two tracks overlap

There's one nuance worth flagging: sometimes a Checkmarx finding is technically a dependency issue that Checkmarx surfaces through dataflow analysis — for example, it traces a vulnerability through a call into a vulnerable version of a library. In that case the finding looks like a custom code issue (because it traces through your code) but the actual fix is a library upgrade (Track 1). Your classifier step needs to distinguish these: if the root cause is a third-party package version, route it to Renovate even if the scanner traced it through your code.

A practical rule: if the remediation action is "change a version number in a manifest file," it's Track 1. If the remediation requires changing your own source code, it's Track 2.
