## How the system works

**Three triggers, one shared pipeline.** Every trigger (build-time, PR merge webhook, daily cron) feeds the same Tekton pipeline — no duplicated logic.

---

## Trigger mechanisms

**Build-time** — add a pre-build Tekton task that runs before your main build. If vulnerabilities are found, the task patches manifests and either fails the build (strict) or auto-commits the fix and retries.

**PR merge webhook** — configure a Bitbucket post-webhook on the `feature → main` event. This fires the Tekton pipeline against the merged commit. If new vulnerabilities are introduced, the pipeline raises a remediation PR against main before the code ships.

**Daily scheduled job** — use Tekton's `CronJob` (or a Kubernetes CronJob that triggers a `PipelineRun`) to scan all repos and branches nightly. This catches drift — cases where a library becomes vulnerable after it was originally pulled in.

---

## The four pipeline tasks

**Task 1 — Clone repos**: Uses Bitbucket's REST API to enumerate repos/branches. For the scheduled job, it iterates all repos. For the other triggers, it operates on the specific repo that fired the event.

**Task 2 — Detect vulnerabilities**: Calls the Wiz and Checkmarx APIs to pull findings, then normalises them into a unified CVE list. This deduplicates common vulnerabilities that appear across all three language stacks.

**Task 3 — Resolve safe versions**: Language-specific sub-tasks run in parallel. Each resolver queries its ecosystem's advisory database for the minimum safe version that satisfies the semver constraints already in the manifest.

| Language | Manifest files | Tools | Advisory source |
|---|---|---|---|
| Java | `pom.xml`, `build.gradle` | Maven Versions Plugin, Gradle Dependency Updates | OSS Index, Sonatype Lift |
| Python | `requirements.txt`, `pyproject.toml`, `Pipfile` | `pip-audit`, Safety CLI | PyPI Advisory DB |
| Node.js | `package.json`, `yarn.lock` | `npm audit fix`, `yarn upgrade` | GitHub Advisory DB |

**Task 4 — Patch & PR**: Modifies the manifest files with safe versions, runs `pip-compile` / `npm install` / `mvn dependency:resolve` to regenerate lock files, then raises a Bitbucket PR with a description that includes CVE IDs, severity, and a link to the original Wiz/Checkmarx finding.

---

## Policy and safety gates

These run between resolution and patching to prevent the pipeline from breaking your apps.

**Upgrade policy engine** — define semver rules per repo. For example: "auto-patch only `patch` and `minor` versions; flag `major` upgrades for human review." Store rules as a YAML config file (`vuln-policy.yaml`) committed to each repo so teams own their own policies.

**Breaking-change gate** — after patching, the pipeline runs your existing test suite. If tests fail, the patch is reverted and the CVE is escalated to the PR description as "manual intervention required." Never silently breaks a build.

**CVE deduplication** — if the same CVE (`log4j`, `requests`, `lodash`) appears in 10 repos, the pipeline raises 10 individual PRs (one per repo) but groups findings in a single audit report so you can track remediation centrally.

---

## Handling custom code vulnerabilities

Wiz and Checkmarx SAST findings for custom code (SQL injection, XSS, hardcoded secrets, etc.) cannot be auto-patched — they require developer judgment. The pipeline handles these by:

1. Attaching the finding with file + line number to the PR description
2. Adding a `vuln:sast` label to the PR so it's clearly flagged
3. Blocking auto-merge on PRs that contain SAST findings (only dependency-only PRs get auto-merged)

## Bitbucket integration specifics

Since you're on **Bitbucket Data Center** (not Cloud), you use the Bitbucket Server REST API:

- `POST /rest/api/1.0/projects/{key}/repos/{slug}/pull-requests` — create PRs
- `POST /rest/api/1.0/projects/{key}/repos/{slug}/webhooks` — register the PR merge webhook
- Use a dedicated service account with write access scoped only to the repos being remediated
- Set branch protection on `main` so the remediation PRs still go through at least one human review unless you're confident in the policy gates

## Recommended rollout order

Start narrow and expand trust gradually:

1. **Week 1-2**: Run pipeline in dry-run/report-only mode. No file changes — just generate the audit report to understand the scale of vulnerabilities.
2. **Week 3-4**: Enable auto-patching for `patch`-level dependency upgrades only, with mandatory human PR review.
3. **Month 2**: Enable `minor` version upgrades with the test-gate in place. Auto-merge only if all tests pass.
4. **Month 3+**: Enable the daily scheduled scanner across all repos. Tune the policy per team's risk tolerance.

The key tools to look at for the resolver layer are **Renovate Bot** (open source, self-hostable, has native Bitbucket Data Center support) and **Dependabot** (GitHub-native but has a self-hosted mode). Renovate in particular handles all three languages, understands monorepos, and can be driven by a config file — it could replace or complement the custom resolver tasks and save significant pipeline development effort.
